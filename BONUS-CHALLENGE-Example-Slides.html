VinUni AI20kPROBLEM · Slide 4/16

🎯 Problem Statement: Tôi đang giải gì?

😱 Pain point

Một startup VN deploy chatbot dùng GPT-4o + Claude Opus 4.7. CEO hỏi:

  • "Hôm qua chúng ta tốn bao nhiêu tiền API?"
  • "User nào đang spam? Tốn nhất?"
  • "Latency p95 có tăng tuần này không?"
  • "Câu trả lời nào bị user thumb-down?"

→ Hiện tại: log file rời rạc, Excel thủ công, không trả lời được.

✅ Success criteria
  • Truy vấn cost/latency/quality theo user/model/route trong <3s
  • Reproducible — query 1 tuần trước phải ra kết quả y hệt
  • Eval score đến trễ (3-7 ngày sau) phải upsert vào row cũ
  • Hỗ trợ time travel để debug "tại sao tuần trước cost đột biến?"
  • Sẵn sàng cho ML team train reward model trên feedback data
📦 Scope (cho project 1 tuần)

1 chatbot, 3 model providers (OpenAI, Anthropic, Cohere), ~100k requests/ngày, 1 region (Singapore). Multi-tenant: 5 customer accounts. Eval = thumbs up/down + LLM-as-judge offline.

Viết problem statement trước khi vẽ kiến trúc là điều tôi thường skip. Nhưng nếu không biết SLA (3s query, 100k req/ngày), tôi sẽ over-engineer — vd dùng Spark cluster thay vì DuckDB.

⚖️ Decision 1: Delta vs Iceberg vs Hudi?

Tiêu chíDelta LakeApache IcebergApache Hudi
Workload chính của tôi Append + MERGE Append + MERGE Upsert-heavy
Engine ưu tiên Spark, DuckDB qua delta-rs Spark, Trino, Flink Spark, Flink
Học ngắn 1 tuần Dễ nhất (Python API mượt) Trung bình Khó nhất
Time travel + RESTORE ✅ RESTORE VERSION AS OF ✅ snapshot id Có nhưng phức tạp hơn
Lab repo dùng sẵn ✅ NB1–NB4 đã dùng Delta Phải tự setup Phải tự setup
Hidden partitioning ❌ Cần explicit ✅ days(ts) Không trực tiếp
🏆 Chọn: Delta Lake (qua delta-rs + DuckDB)

Vì: (1) tôi đã quen từ NB1–NB4, không tốn thời gian học lại; (2) workload "append + MERGE eval scores" hợp Delta; (3) DuckDB query cho cost/latency <3s đủ với 100k req/ngày. Trade-off chấp nhận: mất hidden partitioning của Iceberg → phải explicit partition theo date.

Lúc đầu tôi định chọn Iceberg vì "feature đẹp hơn". Nhưng so với constraint "1 tuần + đã có lab Delta", chọn cái mình quen tốt hơn chọn cái fancy hơn. Engineering = tradeoffs, không phải feature collection.

VinUni AI20kSCHEMA · Slide 7/16

🥉 Bronze Layer: Raw event log

Mục tiêu: NHẬN HẾT, KHÔNG LỌC. Schema permissive, retention 30 ngày, partition theo ngày để vacuum dễ.

-- Path: s3://lh-llm-obs/bronze/llm_events/
-- Format: Delta, partitioned by ingest_date

CREATE TABLE bronze.llm_events (
  event_id        STRING,         -- UUIDv7 từ middleware
  raw_payload     STRING,         -- JSON string nguyên gốc, không parse
  source_topic    STRING,         -- "openai", "anthropic", "cohere"
  ingest_ts       TIMESTAMP,      -- Khi Kafka consumer ghi vào Bronze
  ingest_date     DATE,           -- DERIVED COL cho partitioning
  kafka_offset    BIGINT,         -- Để replay/dedup
  schema_version  STRING          -- "v1" — đổi khi producer breaking change
)
USING DELTA
PARTITIONED BY (ingest_date)
TBLPROPERTIES (
  'delta.logRetentionDuration' = '30 days',
  'delta.deletedFileRetentionDuration' = '30 days'
);
Vì sao chỉ lưu raw JSON?

Tôi không biết tương lai mình cần field nào. Lưu raw → schema downstream có thể evolve mà không cần re-ingest. Đây là Bronze rule: preserve everything.

Common mistake tôi tránh

Không parse JSON ở Bronze. Nếu parse → 1 field NULL/format lạ làm fail toàn batch. Bronze phải là "không bao giờ fail".

Field schema_version là 1 trick tôi học từ slide 23 (Data Contracts). Khi producer thay đổi schema, ta thêm "v2" → consumer biết route khác nhau, không bị silent corruption.

VinUni AI20kSCHEMA · Slide 8/16

🥈 Silver Layer: Cleaned + Joined

-- Path: s3://lh-llm-obs/silver/llm_inferences/
-- Z-ORDER by (tenant_id, model)

CREATE TABLE silver.llm_inferences (
  inference_id      STRING NOT NULL,
  tenant_id         STRING NOT NULL,
  user_id           STRING,
  session_id        STRING,
  request_ts        TIMESTAMP NOT NULL,
  request_date      DATE NOT NULL,            -- Partition col
  provider          STRING,                   -- openai|anthropic|cohere
  model             STRING,                   -- "gpt-4o", "claude-opus-4-7"...
  route             STRING,                   -- "chat", "embed", "tool"
  prompt_tokens     INT,
  completion_tokens INT,
  total_tokens      INT GENERATED ALWAYS AS (prompt_tokens + completion_tokens),
  cost_usd          DECIMAL(10, 6),
  latency_ms        INT,
  status            STRING,                   -- "ok", "rate_limited", "error"
  error_code        STRING,                   -- nullable
  -- Eval cols (đến trễ qua MERGE — initially NULL)
  user_feedback     STRING,                   -- "thumbs_up" | "thumbs_down" | NULL
  judge_score       FLOAT,                    -- 0.0–1.0, từ LLM-as-judge offline
  judge_ts          TIMESTAMP,                -- Khi nào eval được merge vào
  -- Lineage
  bronze_event_id   STRING,                   -- back-pointer to bronze
  silver_run_id     STRING                    -- which dbt run created this row
)
USING DELTA
PARTITIONED BY (request_date);
Z-ORDER chọn (tenant_id, model)

Vì query phổ biến nhất là "cost của tenant X dùng model Y". Z-ORDER co-locate 2 cột này → file pruning > 10× theo lab NB2.

Eval cols nullable

Lúc Silver được tạo (5 phút sau request), eval chưa có. Khi user click 👍/👎, ta MERGE vào row cũ → time travel vẫn xem được "trước khi có eval".

Cột silver_run_id là cách tôi self-document lineage: nếu data sai, tôi join với dbt_runs table biết ngay batch nào lỗi → debug nhanh hơn nhiều so với git blame.

🥇 Gold Layer + MERGE pattern

Gold tables (3 marts)

gold.daily_cost_kpi

tenant_id, model, date, total_cost_usd, request_count, error_rate, p50/p95/p99 latency

gold.user_quality

user_id, week, judge_avg, thumbs_up_rate, thumbs_down_rate, retention_proxy

gold.spam_suspects

user_id, day, req_count, cost_usd, anomaly_score (z-score > 3)

Eval MERGE pattern (key insight)

-- Khi user click 👍/👎, hoặc khi LLM-as-judge batch chạy xong
MERGE INTO silver.llm_inferences AS target
USING staging.eval_updates AS source
  ON target.inference_id = source.inference_id
WHEN MATCHED THEN UPDATE SET
  target.user_feedback = source.user_feedback,
  target.judge_score   = source.judge_score,
  target.judge_ts      = current_timestamp();

-- Sau MERGE, _delta_log ghi version mới
-- → time travel vẫn xem được "trước khi có eval" để so sánh
SELECT * FROM silver.llm_inferences VERSION AS OF 42;
MERGE + time travel cùng nhau cho phép tôi trả lời câu hỏi "liệu eval pipeline mới có gắn judge_score đúng không?" — chỉ cần so sánh VERSION AS OF before với VERSION AS OF after. Không cần snapshot DB thủ công.

⚙️ Production Ops Decisions

📅 Compaction schedule
  • OPTIMIZE daily lúc 03:00 (off-peak)
  • VACUUM 7 ngày retention (compliance team OK)
  • Z-ORDER lại weekly khi data drift
📋 Data contracts
  • Schema version pinned ở producer SDK
  • Breaking change → bump v1→v2 + dual write 2 tuần
  • Great Expectations check ở Bronze→Silver gate
🔍 Lineage
  • OpenLineage emit từ Spark + dbt
  • Marquez UI: trace 1 row Gold ngược về Kafka offset
  • Quan trọng nếu CEO hỏi "số này từ đâu?"
💰 FinOps tier
  • Bronze: S3 Standard 30d → S3 IA → Glacier 1 năm
  • Silver: S3 Standard 90d → IA
  • Gold: S3 Standard mãi mãi (nhỏ, hot)
🚨 Alerting
  • Silver row count < Bronze ×0.95 → alert (drop quá nhiều)
  • Cost daily > baseline ×1.5 → alert (chi tiêu bất thường)
  • p95 latency > 5s sustained → alert
🔐 Compliance (PII)
  • Prompt/response → hash PII trước khi vào Silver
  • Right-to-be-forgotten: DELETE WHERE user_id=X + VACUUM
  • Audit log: ai/khi nào read Gold
Tôi không định viết phần PII ban đầu. Nhưng vì AI20k ở VN — Decree 13/2023/NĐ-CP bắt buộc handle data cá nhân. Nếu skip phần này, kiến trúc không production-ready.

⚖️ Trade-offs (cái tôi cố ý từ chối)

Lựa chọn của tôiCái tôi từ chốiVì sao
5-min micro-batch Bronze→Silver True streaming (Flink) SLA cho dashboard là 15 phút. Streaming overkill, ops cost cao gấp 4×.
Delta Lake Iceberg hidden partitioning Tôi đã quen Delta từ NB1–NB4. 1 tuần không đủ học Iceberg cho production.
DuckDB serve Gold Trino cluster Gold < 1GB. DuckDB 1 node đủ <3s. Trino cần 3+ node, ops phức tạp hơn nhiều.
Hash PII tại Silver Encrypt-at-rest only Hash = irreversible, an toàn hơn. Trade-off: không decrypt được khi cần debug.
Gold rebuild full mỗi ngày Incremental updates Đơn giản hơn, idempotent. Trade-off: tốn compute hơn nhưng <5 phút nên OK.
1 catalog (AWS Glue) Multi-catalog (Glue + Unity + Polaris) Single source of truth. Trade-off: vendor-lock vào AWS, nhưng dev MVP chấp nhận.
Mỗi dòng "không chọn" ở đây đều có thể trở thành future work. Khi scale 10× (1M req/ngày), tôi sẽ revisit từng dòng. Đây gọi là "quyết định reversible" — chọn đơn giản trước, tối ưu sau khi có data thực.

VinUni AI20kPROBLEM SOLVING · Slide 12/16

🔧 Problem solving: 4 failure scenarios

😱 Scenario 1: Kafka topic bị xóa nhầm

Hậu quả: Mất raw events giữa T1 và T2.

Giải pháp: Bronze đã ghi xuống S3 → an toàn. Re-process từ Bronze. kafka_offset giúp dedup khi producer replay.

😱 Scenario 2: Eval pipeline bug ghi sai score

Hậu quả: 10k row có judge_score sai.

Giải pháp: RESTORE TABLE silver.llm_inferences VERSION AS OF X (version trước MERGE). Time travel cứu mạng.

😱 Scenario 3: User yêu cầu xóa dữ liệu (Decree-13)

Hậu quả: Phải xóa physically (không soft-delete).

Giải pháp: DELETE WHERE user_id=X + VACUUM RETAIN 0 HOURS (cẩn thận: tắt time travel cho table này nếu cần).

😱 Scenario 4: Cost spike đột ngột (CEO panic)

Hậu quả: Cần biết ngay user/model nào.

Giải pháp: Query Gold spam_suspects → list top user có anomaly_score > 3 trong 24h. Lineage trace ngược Bronze để xem prompt thực.

Đặt câu hỏi "Cái gì có thể đi sai?" sớm giúp tôi thiết kế cho recovery, không chỉ cho happy path. Kafka offset, MERGE rollback, VACUUM 0h — những detail này không hiện trên diagram nhưng quyết định "production-ready" vs "demo-ware".

SELECT date, SUM(cost_usd) AS daily_cost
FROM gold.daily_cost_kpi

🎓 Lessons Learned (góc nhìn sinh viên)

✅ Cái đã đúng
  • Chọn problem theo decision matrix → tránh paralysis
  • Viết problem statement trước kiến trúc
  • Reuse Delta từ lab → tiết kiệm 2 ngày học format mới
  • Buffer day D7 → kịp deadline dù D6 trễ
  • Mô tả từng "không chọn" trong trade-offs slide
⚠️ Cái cần làm khác lần sau
  • Lúc đầu định dùng Iceberg cho "đẹp" — phí 1 buổi research mà không apply
  • Schema Bronze ban đầu thiếu schema_version → bài học từ slide 23
  • Quên PII handling đến scenario 3 mới nhớ — phải gắn từ đầu
🔮 Future work nếu scale 10×
  • Migrate Delta → Iceberg để có hidden partitioning
  • Trino cluster thay DuckDB single-node
  • Real streaming (Flink) thay micro-batch
  • Multi-region (HCM + HN) với active-active
🧠 Insight về môn học
  • "Lakehouse" không phải 1 tool — là tập hợp các quyết định
  • Medallion là khuôn mẫu suy nghĩ, không phải obligation
  • ACID trên S3 = "magic" nhờ _delta_log JSON files
  • Production = handle 4 failure modes, không chỉ happy path
Quan trọng nhất tôi học: kiến trúc tốt = ít opinion hơn, nhiều constraint hơn. Khi tôi viết "tại sao chọn Delta", tôi list constraint (1 tuần, đã quen, 100k req/ngày) — không list opinion ("Delta xịn hơn"). Constraint khách quan, opinion chủ quan.
